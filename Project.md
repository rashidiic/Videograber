# Project.md

## Описание проекта

**InstaSoundSaveBot** — это Telegram-бот на Python, который принимает ссылку на видео, скачивает медиафайл, извлекает из него аудио в формате MP3, отправляет файл на внешний HTTP endpoint и возвращает пользователю красиво оформленный итоговый отчет.

Сейчас проект работает как **пайплайн обработки**:

1. Пользователь отправляет сообщение с URL.
2. Бот находит и валидирует ссылку.
3. Бот скачивает исходное медиа через `yt-dlp`.
4. Бот конвертирует файл в MP3 через `ffmpeg`.
5. Бот отправляет MP3 в HTTP API.
6. Бот возвращает результат в Telegram в читабельном формате с эмодзи и HTML-разметкой.

Важно: на текущем этапе ответ API частично замокан. Реальная загрузка файла происходит, но итоговые данные для ответа берутся из mock-слоя.

---

## Как устроен проект

Проект разделён на несколько логических слоёв:

- `src/bot/` — запуск Telegram-бота, конфигурация, логирование и обработчики команд.
- `src/services/` — сервисы скачивания, конвертации и отправки данных.
- `src/utils/` — вспомогательные функции для URL, форматирования текста и работы с временными файлами.
- `src/tests/` — тесты для ключевых частей логики.
- Корневые файлы — Docker, зависимости, запуск и описание проекта.

## Точка входа и запуск

### `src/bot/__main__.py`

Файл позволяет запускать бота командой:

```bash
python -m bot
```

Он просто импортирует `main()` из `src/bot/main.py` и вызывает её.

### `src/bot/main.py`

Это главный файл запуска приложения. Он делает следующее:

- создаёт объект настроек через `Settings()`;
- настраивает логирование;
- создаёт папку для временных файлов;
- собирает Telegram Application через `ApplicationBuilder`;
- регистрирует все обработчики сообщений;
- подключает глобальный обработчик ошибок;
- задаёт команды меню Telegram через `BotCommand`;
- запускает polling.

Дополнительно здесь определена асинхронная функция `setup_menu_commands()`, которая устанавливает команды меню:

- `/start` — запуск и приветствие;
- `/help` — справка по использованию.

### `src/bot/logging_config.py`

Отвечает за стандартную конфигурацию логов:

- формат логов задаётся через `logging.basicConfig()`;
- вывод идёт в `stdout`;
- уровень логирования берётся из настроек.

Также здесь есть `get_logger(name)`, чтобы другие модули получали свой logger через `logging.getLogger()`.

### `src/bot/config.py`

Файл содержит конфигурацию на базе `pydantic-settings`.

Класс `Settings` читает параметры из `.env` и умеет брать значения по умолчанию:

- `telegram_token` — токен Telegram-бота;
- `dummy_api_url` — URL для загрузки MP3;
- `tmp_dir` — папка для временных файлов;
- `log_level` — уровень логов.

Если переменная окружения не задана, используются дефолтные значения.

## Обработчики Telegram

### `src/bot/handlers/__init__.py`

Этот файл собирает все обработчики в одном месте и регистрирует их в приложении.

Сейчас он добавляет:

- `/start`;
- `/help`;
- обработчик обычного текстового сообщения с URL.

Это удобно, потому что вся регистрация маршрутов Telegram находится в одном месте.

### `src/bot/handlers/start.py`

Здесь находятся команды приветствия и справки.

Что делает файл:

- хранит текст помощи в `HELP_TEXT`;
- отвечает на `/start`;
- отвечает на `/help`;
- отправляет пользователю более живое приветствие с эмодзи и HTML-форматированием.

Сейчас тексты специально сделаны дружелюбнее:

- есть эмодзи;
- подсказка написана списком;
- пример ссылки показывается обычным текстом, чтобы Telegram мог показать preview видео;
- сообщение отправляется с `parse_mode="HTML"` там, где это нужно.

### `src/bot/handlers/url_handler.py`

Это основной обработчик бота. Он работает с обычными текстовыми сообщениями, если это не команда.

Логика такая:

1. Берётся текст сообщения.
2. Через `extract_url()` из текста достаётся первая подходящая ссылка.
3. Если ссылка не найдена, бот показывает подсказку.
4. Если ссылка невалидна, бот просит отправить нормальный `http://` или `https://` URL.
5. Если URL подходит, бот:
   - сообщает, что начал скачивание;
   - скачивает видео через `download_video()`;
   - конвертирует файл в MP3 через `convert_to_mp3()`;
   - отправляет MP3 через `DummyApiClient.upload_mp3()`;
   - форматирует итог через `format_response()`;
   - отвечает пользователю.

Дополнительно:

- используются `try/except` на каждом этапе пайплайна;
- ошибки показываются пользователю в более мягком виде;
- в лог пишется подробная техническая информация;
- в конце вызывается `TemporaryFileManager.cleanup()`.

### `src/bot/handlers/errors.py`

Глобальный обработчик непойманных исключений.

Если где-то внутри Telegram runtime случается исключение, этот обработчик:

- пишет ошибку в лог;
- пытается отправить пользователю короткое безопасное сообщение;
- не даёт боту упасть без объяснения.

## Сервисы обработки медиа

### `src/services/downloader.py`

Сервис скачивания видео/аудио через `yt-dlp`.

Как работает:

- создаёт настройки `YoutubeDL`;
- использует шаблон имени файла вида `%(id)s.%(ext)s`;
- запускает скачивание в отдельном потоке через `asyncio.get_running_loop().run_in_executor()`;
- после завершения проверяет, что файл реально появился на диске;
- пишет лог с размером файла.

Отдельная функция `_human_size()` используется для удобного отображения размера файла в логах.

### `src/services/converter.py`

Сервис конвертации в MP3.

Он вызывает `ffmpeg` через `subprocess.run()` и делает это тоже в отдельном потоке, чтобы не блокировать event loop.

Параметры конвертации:

- без видео дорожки (`-vn`);
- кодек `libmp3lame`;
- битрейт `192k`;
- частота `44100`;
- 2 канала.

После конвертации проверяется, что MP3-файл действительно создан.

### `src/services/api_client.py`

Клиент для загрузки MP3 на внешний HTTP endpoint.

Сейчас его логика такая:

- создаётся `httpx.AsyncClient`;
- MP3 отправляется как multipart form upload;
- HTTP-ошибки не считаются фатальными для всей задачи;
- после запроса возвращается мок-результат из `get_mock_response()`.

То есть загрузка существует, но итог ответа формируется не на основе реального API-ответа, а на основе заготовки.

### `src/services/mock_data.py`

Содержит функцию `get_mock_response()`.

Она возвращает словарь с примером результата обработки, включая:

- `status`;
- `filename`;
- `duration_seconds`;
- `language`;
- `confidence`;
- `transcription`;
- `sample_rate_hz`;
- `bitrate_kbps`;
- `channels`.

Этот модуль полезен как временная подмена реального speech-to-text ответа.

## Утилиты

### `src/utils/url_validator.py`

Этот файл отвечает за поиск и проверку ссылок.

Есть две функции:

- `extract_url(text)` — ищет первую ссылку в тексте через regex;
- `is_valid_url(url)` — проверяет, что URL имеет схему `http` или `https` и содержит `netloc`.

Именно этот модуль решает, можно ли вообще отправлять сообщение дальше по пайплайну.

### `src/utils/formatters.py`

Формирует итоговое сообщение пользователю.

Сейчас функция `format_response(data)`:

- строит многострочный ответ;
- добавляет эмодзи;
- использует HTML-разметку для выделения ключевых полей;
- красиво показывает:
  - файл;
  - длительность;
  - язык;
  - confidence;
  - sample rate;
  - bitrate;
  - channels;
  - transcription.

Здесь же решается, как именно выглядят сообщения в Telegram. Именно этот файл отвечает за тот формат, который пользователь видит в конце обработки.

### `src/utils/file_utils.py`

Содержит `TemporaryFileManager`.

Он задуман как менеджер временных файлов:

- `register(path)` — запомнить файл для удаления;
- `cleanup()` — удалить все зарегистрированные файлы.

Сейчас этот менеджер создаётся в обработчике URL и вызывается в `finally`, но в текущей реализации файлы почти не регистрируются, поэтому эта часть пока работает не на полную силу.

## Конфигурация и запуск в Docker

### `Dockerfile`

Проект собирается в два этапа:

1. **builder**
   - ставится `uv`;
   - устанавливаются зависимости;
   - копируется исходный код;
   - собирается виртуальное окружение.

2. **runtime**
   - ставится `ffmpeg` и сертификаты;
   - создаётся системный пользователь `app`;
   - копируется готовое окружение и исходники;
   - настраиваются переменные окружения;
   - контейнер запускает `python -m bot`.

Это хороший подход, потому что runtime-образ получается чище и меньше.

### `docker-compose.yaml`

Определяет сервис `bot`.

Здесь настроено:

- сборка из `Dockerfile`;
- имя image `telegram-video-mp3-bot:latest`;
- контейнер `telegram-video-mp3-bot`;
- `restart: unless-stopped`;
- подключение `.env`;
- прокидывание `TMP_DIR` и `LOG_LEVEL`;
- volume для временных файлов;
- healthcheck;
- логирование через `json-file` с ротацией.

### `Makefile`

Удобные команды для разработки и Docker:

- `make run` — запуск бота локально;
- `make test` — тесты;
- `make lint` — `ruff check`;
- `make typecheck` — `mypy`;
- `make format` — автоформатирование;
- `make docker-build` — сборка Docker-образа;
- `make docker-up` — пересборка и запуск контейнера;
- `make docker-down` — остановка;
- `make docker-logs` — просмотр логов.

Важно: в `Makefile` отступы должны быть TAB, а не пробелы.

## Поток работы бота

Полная цепочка выглядит так:

1. Пользователь отправляет ссылку в Telegram.
2. `URLHandler` достаёт URL из текста.
3. `url_validator` проверяет корректность ссылки.
4. `downloader` скачивает медиа.
5. `converter` переводит файл в MP3.
6. `api_client` отправляет MP3 на HTTP endpoint.
7. `mock_data` даёт структуру ответа.
8. `formatters` превращает данные в красивое сообщение.
9. Бот отправляет финальный ответ пользователю.

## Что уже хорошо, а что пока упрощено

### Уже реализовано

- Telegram polling.
- Команды `/start` и `/help`.
- Меню команд в Telegram.
- Скачивание медиа.
- Конвертация в MP3.
- Форматированный ответ пользователю.
- Обработка ошибок на каждом этапе.
- Логирование.
- Docker-запуск.

### Пока упрощено или замокано

- Реальный ответ speech-to-text API не используется.
- `upload_mp3()` возвращает mock-данные вместо реального результата API.
- Очистка временных файлов пока не полностью завязана на регистрацию путей.

## Итог

Этот проект — Telegram-бот с понятной модульной архитектурой:

- бот-слой отвечает за Telegram;
- services-слой отвечает за обработку медиа;
- utils-слой отвечает за вспомогательную логику;
- tests-слой фиксирует ожидаемое поведение;
- Docker и Makefile делают запуск и деплой удобными.

Если позже появится настоящий speech-to-text backend, его логичнее всего подключать в `src/services/api_client.py` и, при необходимости, расширять `src/utils/formatters.py` под новый формат ответа.
